## 目的

`docs/` 配下の仕様と変更されたコードを突き合わせ、**仕様と実装のずれ**を指摘する。

## スコア（0–100）

| 範囲   | 説明                                                                                       |
| ------ | ------------------------------------------------------------------------------------------ |
| 90–100 | 🔴 **Critical** — 仕様と実装が明文化された事実として矛盾（どちらかを直さないと整合しない） |
| 75–89  | 🟡 **Important** — 仕様の読みはほぼ一択だが、表現や周辺文脈でわずかに揺れる                |
| 50–74  | 🟢 **Low** — 解釈の余地や粒度不足があり、前提を明記すれば指摘できる軽微なずれ              |
| 0–49   | 出力に含めない（推測が多い・両方読みうる）                                                 |

**閾値:** **スコア 50 未満**の指摘は出力に含めない。

## 出力形式

### 各指摘

- **スコア**: 整数（50–100）
- **分類**: `Critical` / `Important` / `Low`（🔴 / 🟡 / 🟢）
- **仕様書**: `docs/specs/...` など根拠パス
- **実装**: `パス:行番号`
- **Issue**: ずれの内容
- **Suggestion**: `仕様更新` / `実装修正` / `両方` / `要判断` のいずれかと、具体案

分類は **Critical** → **Important** → **Low** の順、同順位なら **スコア** の高い順。指摘ゼロなら「✅ 報告すべき指摘はありません。」のみ。
