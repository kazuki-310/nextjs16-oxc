# 認証要件（Auth.js）

## 方針

- **Auth.js** を採用する（Credentials プロバイダ中心）。
- ログイン後のセッションは **`session.strategy: "jwt"`** で **JWT を Cookie に載せて管理**する。

## 機能要件

### ログイン

- **`nickname` または `email` と `password`** でログインできる。
- ユーザーによっては **`nickname` のみ**、**`email` のみ**のパターンがある（どちらか一方が無い場合がある）。
- ログイン時は **`authorize` 内で識別子を解決**する（例: `@` の有無、`nickname` / `email` で `findUnique`）。

### フォーム・バリデーション（クライアント）

- **必須ではないが推奨**: trim、文字数、**ニックネーム欄に `@` を含めない**など、**UX と無駄なリクエスト削減**のためクライアントでもバリデーションしてよい。
- **サーバーが正**: クライアントは改ざん可能なので、**`authorize` 側でも同じルールを検証**する（識別子の解釈はサーバーで一本化）。

### 登録

- 要件に沿った登録フロー（識別子の一意性、パスワードポリシーなどは実装時に定義）。

## セキュリティ要件

### パスワード

- **平文で DB に保存しない**。**bcrypt** でハッシュ化して保存する（コストファクターは実装時に決定）。
- 登録時にハッシュ化し、ログイン時は **`authorize` 内で照合**する（Auth.js は Credentials のハッシュを自動では行わない）。

#### ハッシュ強度とブルートフォース対策の関係

- **bcrypt / Argon2 などは「1 回の照合のコスト」を上げる**ためのもので、**ログイン API への大量試行（オンライン総当たり）を止める主役ではない**。
- **連続失敗回数・ロックは DB で管理**し、`authorize` で制御する（下記）。**ハッシュアルゴリズムを変えても、失敗カウントなしではブルートフォース対策として不十分**。

### ログイン失敗とロック（DB で管理・確定）

別プロジェクトで行っていた **「失敗回数に応じたロック」** と同様に、**サーバー側で DB を更新して制御**する。

- **JWT に失敗回数を載せない**（クレームはユーザー識別・期限などに留める）。
- **同時ログイン数・セッション数の上限**が要件に含まれる場合は、**別途**（`jti` + 失効テーブル、セッション行、など）を設計する。本要件の必須項目は **ログイン失敗のブルートフォース対策** とする。

#### DB に持たせる項目（案）

`employees`（または認証に使うユーザーテーブル）に、少なくとも次を追加する想定とする。カラム名は実装時に Prisma に合わせてよい。

| 目的           | 例（カラム名）          | 説明                                                      |
| -------------- | ----------------------- | --------------------------------------------------------- |
| 連続失敗回数   | `failed_login_attempts` | パスワード誤りの連続回数。ログイン成功時に 0 にリセット。 |
| ロック解除時刻 | `unlock_at`             | ロック中はこの時刻までログイン不可。未ロックなら `NULL`。 |

運用ルールの例（数値は設定可能にする想定）:

- 失敗のたびに `failed_login_attempts` を増やす。
- **上限回数に達したら**一定時間ロックする（例: **9 回目の失敗後に 15 分ロック** — 先に共有した既存実装のイメージ。閾値・時間は要調整）。
- ロック中は **`authorize` の早い段階**で拒否し、パスワード照合まで進めない。
- **ログイン成功時**は `failed_login_attempts` をリセットし、`unlock_at` をクリアする。

#### IP ベースのレート制限

- **当面は導入しない**（実装を単純化する）。
- **トレードオフ**: 同一 IP から複数アカウントへ順に試す攻撃などへの耐性は、**DB のアカウントロックだけより弱くなりやすい**。負荷やインシデントを見て、**後から** Middleware / Redis / インフラで足す選択肢を残す。

### 列挙耐性

- 「ユーザーが存在しない」と「パスワードが違う」で、**利用者に分かる差が出ない**ようメッセージ・レスポンスを揃える（実装時に詳細化）。

## Auth.js 実装メモ（要点）

- **`CredentialsProvider` の `authorize`**: 識別子解決 → **ロック確認（DB）** → パスワード照合 → 失敗時は失敗カウント更新、成功時はリセット。
- **`session.strategy: "jwt"`**: ログイン成功後に JWT を Cookie に載せる。以降は署名・`exp` で検証（**毎回パスワード照合ではない**）。

- bcrypt のコストファクター。
- パスワードポリシー（最低文字数など）。
